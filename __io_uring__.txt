一些值得注意的点

trace __n_total_submit_
trace __n_newly_submit_
atomic ref
IORING_SQ_CQ_OVERFLOW  back-pressure 
cq_entries 为 sq_entries 的 2~4 倍；或者 IORING_SETUP_CQ_NODROP
连接关闭但还有 pending recv/send → 必须先 ASYNC_CANCEL 再 close
Buffer 被回收到池子并重新分配 → 但旧 CQE 还没回来 → 数据覆盖
sqe->user_data index/generation 或 intrusive refcount/packed struct
res == 0 对 recv → 对端关闭（不是错误！）
res == -ECANCELED → 被 ASYNC_CANCEL 取消的，正常路径
res == -EAGAIN → SQPOLL 模式下可能出现，需要重试）（塞回去，IORING_ENTER_SQ_WAKEUP ）
res == -ENOBUFS → buffer ring 空了，所有 buffer 都在用
RPC 框架要做 feature probing：io_uring_probe 探测支持的 op，graceful fallback。
Connection 上可能同时有：
  - 1 个 pending RECV
  - 1 个 pending SEND  
  - 1 个 pending TIMEOUT

close 时必须等所有 CQE 都回来（或 cancel 成功）才能销毁 Connection。
方案：每提交一个 SQE → ref_count++；每收到一个 CQE → ref_count--；减到 0 才真正 destroy。
